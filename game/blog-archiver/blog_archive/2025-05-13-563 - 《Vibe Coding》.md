# 563 - 《Vibe Coding》

![](https://cdn.jsdelivr.net/gh/sorrycc-bot/image-2025-04@main/uPic/vibecoding.png)

上周体验了一下比较纯粹的 Vibe Coding。也在群里面做了一些记录，然后在这里就再聊一下我关于这一部分的一个方法和感受。也聊 Vibe Coding 。

1、本次产出的是一个 npm 库 [upgear](https://github.com/sorrycc/upgear)，用来做 cli 自动更新的，已在 takumi 里用上。AI 产出的整体代码质量挺高，而且考虑的非常全面，至少我自己是不会一下子考虑到这么全的。甚至有些考虑地太全面了，比如安全校验方面的，我担心代码会变得太复杂，就没让加上。

2、什么是 Vibe Coding ？中文翻译有多种，比如氛围编程、随性编程、随缘编程、直觉编码、自发编码，你喜欢哪种？Vibe Coding 的概念由 Andrej Karpathy 于 2023 年 2 月提出，其定义是「指人类通过语音或文本向 AI 发出指令，由 AI 主导编写代码的一种编码风格」。在这一过程中，自己几乎不直接编写代码，而是通过与 AI 的交互完成从需求定义到最终应用的开发。

3、对于简单的任务来说，您可以直接说命令然后要求 AI 帮你完成。但对于一些复杂任务来说，就需要有一些策略，不能太直接。**简而言之，我的策略是「原始需求」>「需求文档」>「设计文档」>「任务拆解文档」>「让 AI Step by Step 实施」。** 我在 [upgear 的 context/init](https://github.com/sorrycc/upgear/tree/master/context/init) 目录下保留了这部分文档，供参考。

1）原始需求 > 需求文档。

方法是和 ChatGPT 4o 聊天，要求产出 [spec.md](http://spec.md) 。格式可以自己定义。同时要求他「请帮我想的更全面一些。你先不要着急帮我产出文档，为了质量更高的答案，可以要求我补充更多信息。」

记得完成后找 o3 或其他 think model 再审查一遍，补充细节。

2）需求文档 > 设计文档。

方法也是和 ChatGPT 4o 或 Gemini Pro 聊天，附上代码，要求产出 [design-doc.md](http://design-doc.md) 。同时要求他「不编写具体的实现代码，仅专注于设计实现需求的最佳方法。」

3）需求文档 + 设计文档 > 任务拆解文档。

略。

4）Step by Step 实施。

![](https://cdn.jsdelivr.net/gh/sorrycc-bot/image-2025-04@main/uPic/Cursor%202025-05-09%2017.07.50.png)

也可以让 Cursor 执行「implement all the tasks」，然后去干其他事，过十几二十分钟再来看看。需要注意的是，这一步得用 Claude 3.x Sonnet，保证代码质量。

4、再补充一些 Tips。

1）每一步得人工确认，别 AI 产出啥就是啥。。如果你对 AI 产出的方案很陌生，后续产出的代码也会感觉很陌生。可能会导致最终变成无法维护的代码。

2）利用 Cursor 的 Custom Mode 定义，把上述步骤结合进去，让整体操作更流畅一些。注：Takumi 后续会内置这套流程（可能作为插件？）。
