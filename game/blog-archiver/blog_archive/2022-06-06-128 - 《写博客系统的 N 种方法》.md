# 128 - 《写博客系统的 N 种方法》

有人热衷于写博客，有人热衷于写博客系统，这是完全不同的两种类型的乐趣。尽管有不少同学在写完博客系统之后可能从来没有更过一篇博客，但是「写博客前先写个博客系统」，并不丢人。这是演练新技术非常好地试验田，比如 SWYX 大佬从 2017 到 2022 一共换了 [6 套博客系统的方案](https://www.swyx.io/rewrite-2022)。

以下是一些选择，

1、懒得折腾型，推荐直接用 [https://ghost.org/](https://ghost.org/) 或 [https://wordpress.com/](https://wordpress.com/) ，应该是最省事的，适用面也最广。

2、全栈型推荐 [Remix Indie Stack](https://github.com/remix-run/indie-stack)，部署在 [Fly.io](http://Fly.io)，数据库用 [Fly.io](http://Fly.io) 的 PostgreSQL 或 PlanetScale 提供的数据库服务，再加个 Redis 服务做缓存。

3、[oldj](https://oldj.net/article/2022/06/05/tech-of-this-blog-2022/) 在 2022 年采取的最新方式是 Next.js + Django + PostgreSQL，Django 提供类 Headless CMS 的服务，具体内容交给 Next.js 渲染，静态输出，SEO 友好。

4、我的[个人网站](sorrycc.com)新官网计划的技术栈是 Strapi + Heroku + Umi + PlanetScale + Vercel。Strapi 是目前 star 数最高的开源 Headless CMS，可以 self host，我部了一个在 Heroku 上；Umi 作为前端应用，通过 api routes 访问 Strapi 的 API；同时通过 PlanetScale 加了一层 redis 缓存提速，缓存的失效和更新通过 Strapi 的 webhook 来触发。

5、[SWYX](https://www.swyx.io/rewrite-2022) 在 2022 年最新的方案是 Sveltekit + TailwindCSS 3 + Github ISSUE + Netlify。Github ISSUE 作为 CMS 和 评论内容来源。
