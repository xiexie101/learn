# 78 - 《新老板的几个鼓励》

换组了，新老板分享了他三个鼓励团队同学做的事，摘其中有感触的两点。

1、目标导向做事

围绕目标做事，确保动作有效性。目标就在那，有人走直线，有人走曲线，有人走不到。尤其是对于做基础技术的同学来说，经常会发现一个新奇的点，然后还没等想明白，就投时间进去，做大量无用功。

我在做工具和框架时也做过不少无用功。比如 umi ui 的可视化辅助编程，花了大量精力和人力，结果开发出来没有人用，推广不开；比如组件、区块、模板的概念，推出模板后，结果发现现实情况下大家很少用。

解法是围绕问题和目标做事。投入之前多讨论，多问一句，他解的是什么问题，和目标是否相关。举一些例子。比如下午和同事聊 umi lint 的技术方案，lint 存在规则不敢更新的问题，之前定的技术方案是通过存 meta + diff + 文件级 git 信息查询的方式做，听着就很复杂，然后聊完后发现新方案本身是白名单机制，变更几率少且可控，规则不敢更新的问题并不存在，然后过段地把这部分投入砍掉；比如之前看到拍平的约定式路由结构我很想做，然后聊的时候同事问了句「他解什么问题」，突然发现好像优先级也并不高，然后就过段扔到「Might Do」列表里。

每个人的精力有限，决定不做什么通常可以节约大量时间，留给目标任务，还可以让 KPI 更好看。

很多人卷和加班，是因为没能走直线，虽然看起来很忙，但本质上却是在做无用功。2020 年我做双 12 大促时，也很少需要加班。

当然不是说看不透的就不投入，基础技术通常会留一个百分比的精力做面向未来的事，并且不做或做很少的 KPI 要求。比如我们去年（2021）做 esmi，面向未来的 esm bundless 方案，虽然做出来了，但由于服务端性能问题，没有推给业务线使用。

2、以项目制跟进事项

有规划、设计、时间安排、过程跟进和总结，会让大家的节奏感更好。

这是比较困扰我的一个点，基础技术通常比较「随意」，拍的时间点经常延期。比如大的版本，Umi 4 延了 1 个月，Father 4 延了数月还没开工，Dva 3 延了几年也还没开工。

有一些原因吧。比如 1）没有配备 PM / PD，我今年有计划学下 PMP 课程，2）专业度问题，评估的时间不准，需要两个月的评估了 1 个月，延期风险就很高了，3）RFC 没讨论清楚，边做边想，做着做着发现任务增加了 1 倍。

今年准备试下更为正规的项目制，让延期更少发生。
