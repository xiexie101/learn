# 508 - 《用 Cursor 半自动地整理资讯》

1、Cursor Chat 的量基本用不完，于是想着干点啥。

![](https://tcc.sorrycc.com/p/95b60943-01e6-4c6d-987f-f30664a52809.png)

2、我个人更倾向于 [Local First](https://publish.obsidian.md/andgate/posts/Local-First) + 纯文本的，把所有数据都掌握在自己手里，心里感觉踏实。就像我更倾向于用 Obsidian 而不是 Notion 一样。感觉围绕 Cursor 的非编码用法也可以有很多，比如整理资讯、翻译文章、总结、转换、提取、查找错别字等等。

3、首先我把「整理前端资讯」这件事挪到了 Cursor。也想过要不要搞成全自动，感觉一是成本比较高，二是全自动会让自己无感，把自己摘掉并不是件好事，还是得多少参与一点，就像玩游戏如果开了挂，爽了一时，但之后就会索然无味。

我打了个仓库，目录结构如下。`wip` 是待整理的 markdown 文件，`wip.md` 是整理后的文章，`extract_list.txt` 是提取文章列表的 prompt，`translate.txt` 是翻译文章的 prompt。

```
+ wip
- wip.md
- extract_list.txt
- translate.txt
```

我的主要信息源之一是一些 weekly 的 newsletter，比如 [React Digest](https://reactdigest.net/)、[Bytes](https://bytes.dev/)、[Cassidoo](https://cassidoo.co/) 等，参考 [268 - 《我的 RSS 订阅》](https://sorrycc.com/rss-subscriptions)。

当有新的 weekly 出现，我会。1）先用 [MarkDownload](https://chromewebstore.google.com/detail/markdownload-markdown-web/pcmpcfapbekmbjjkdalcgopdkipoggdi) 手动保存到本地，或者执行脚本用保存到本地，背后是 PlayWright + [MarkitDown](https://github.com/microsoft/markitdown) / [Markify](https://github.com/sorrycc/markify)，2）然后在 Cursor Chat 里执行 `@extract_list.txt` 提取文字列表。见图。

![](https://tcc.sorrycc.com/p/5822d423-9a67-4d50-b363-9a509b165b23.gif)

prompt 如下。

```txt
- 请整理 @wip 下所有 markdown 文件里的文章到 wip.md
- 排除明显的推广文章
- 包含文章、链接和介绍，以文件名分组
- 不要遗漏文件和文章
```

然后针对一些好的文章，我会。1）用前面的脚本把文章的 Markdown 格式保存到本地，2）在 Cursor Chat 里执行 `@translate.txt` 把文章翻译成中文。

![](https://tcc.sorrycc.com/p/3e217770-a542-4ff1-9aa5-7d0b95aa6a26.png)

prompt 如下。

```txt
你是一名专业的前端开发者，熟悉 React、Node、Vue、Angular 等技术栈，请做这些事。

- 把这篇文章翻译成中文，注意盘古之白（中文和字母，中文和数字之间需要空格）
- 一次性创建完整翻译，不要问是否继续
- 在最前面加一段总结，100 - 300 字，不要超过 300 字。不要换行，如有列表项需要换行，请以 1), 2) 的方式让他归为一类，总结里不要出现多行代码。总结前面加「编者注：」，同时前后加「**」做高亮。不要用太官方的口吻。
- 对你觉得比较重要的句子加高亮，高亮的方式是在前后加上 ==，比如原来是「foo」，高亮后是「==foo==」
- 不要删除文件头部的「原文、作者和译者」的部分
- 代码中如果出现连续两个空行，请只保留一个空行
- 保存成一个新文件，文件名用「译：{中文标题}.md」，中文标题为原英文标题翻译城中文后的内容
- 修改前面的「译者：ChatGPT 4 Turbo」为「译者：Claude 3.5 Sonnet」
- 文末加一个空行
```

4、一些注意点：

1）偶尔会遇到乱码，最近好像没了？  
2）太长的文章会 cancel，可以尝试把「一次性创建完整翻译，不要问是否继续」这句去掉，一段段翻译  
3）太长的文章翻译体验还是不太好，我会 fallback 回之前基于 ai + 段落拆分的脚本
