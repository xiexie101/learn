# 译：Superpowers：我在 2025.10 如何使用 Code Agent

> 原文： [https://blog.fsck.com/2025/10/09/superpowers/](https://blog.fsck.com/2025/10/09/superpowers/)  
> 作者： Jesse Vincent  
> 译者： Gemini 2.5 Pro

感觉我几天前才写了《[我如何在 2025 年 9 月使用编程代理](https://blog.fsck.com/2025/10/05/how-im-using-coding-agents-in-september-2025/)》这篇文章。

在那篇文章的开头，我提到我的工作流程从那时起又有了一些演进。

过去几周，我一直在开发一套工具，用来更好地提炼和系统化我的工作流程，并帮助我更好地引导我的代理伙伴。我本打算这个周末开始为这个系统编写文档，但今天早上，Anthropic 突然为 claude code 推出了一个[插件系统](https://docs.claude.com/en/docs/claude-code/plugins)。

如果你不想继续读下去，想直接上手我的新玩具，它们已经足够自动化了。你需要 Claude Code 2.0.13 或更高版本。启动它，然后运行：

```
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace
```

退出并重启 `claude` 后，你会看到一个新注入的提示：

```
<session-start-hook><EXTREMELY_IMPORTANT>
You have Superpowers.

**RIGHT NOW, go read**: @/Users/jesse/.claude/plugins/cache/Superpowers/skills/getting-started/SKILL.md
</EXTREMELY_IMPORTANT></session-start-hook>
```

这就是启动 `Superpowers` 的引导程序。它教会了 Claude 几件重要的事：

1.  You have skills. They give you Superpowers.
2.  Search for skills by running a script and use skills by reading them and doing what they say.
3.  If you have a skill to do something, you _must_ use it to do that activity.
4.  你拥有技能。它们赋予你超能力。
5.  通过运行脚本来搜索技能，通过阅读技能并按其说明操作来使用技能。
6.  如果你有某个技能来做某件事，你_必须_使用它来完成这项活动。

# 编码工作流 [#](#the-coding-workflow)

它还内置了我[之前写过](https://blog.fsck.com/2025/10/05/how-im-using-coding-agents-in-september-2025/)的“头脑风暴 -> 计划 -> 实现”的工作流。最大的变化是，你不再需要运行命令或粘贴提示。如果 Claude 认为你正试图开始一个项目或任务，它_应该_会默认先与你讨论计划，然后再开始执行。

头脑风暴结束后，如果你在一个 git 仓库中，它会自动为项目创建一个 worktree 并切换到该目录。这意味着你可以在同一个项目上并行启动多个任务，而不会互相干扰。

然后它会让你选择：

1.  last month’s process (where you’d open a second `claude` session and act as a human PM for the architect and implementer.)
2.  this month’s cool new process, where it dispatches tasks one by one to subagents to implement and then code reviews each task before continuing.
3.  上个月的流程（你需要打开第二个 `claude` 会话，扮演人类项目经理的角色，指挥架构师和实现者）。
4.  这个月酷炫的新流程，它会逐一将任务分派给子代理去实现，然后在继续之前对每个任务进行代码审查。

无论哪种方式，Claude 都会实践红/绿 TDD（测试驱动开发），即先写一个失败的测试，然后只编写刚好能让测试通过的代码，接着继续下一步。

在实现过程的最后，Claude 现在会提议创建一个 GitHub pull request，将 worktree 合并回本地的源分支，或者就此停止。

但这些都不是最有趣的部分。

# 有趣的部分 [#](#the-interesting-part)

技能才是有趣的部分。而且在不远的将来，你会从……几乎所有人那里听到更多关于它们的消息。

技能，就是赋予你的代理超能力的东西。

它们第一次真正引起我的注意是在几周前，当时 Anthropic 推出了改进的 Office 文档创建功能。当这个功能发布时，我到处看了看——我让 Claude 告诉我它所有的新技能。而它也[非常乐意地全盘托出](https://claude.ai/share/0fe5a9c0-4e5a-42a1-9df7-c5b7636dad92)。

从那以后，我开始到处看到类似技能的东西。

几周前的一个周五，我看到一个非常酷的技术演示，他们谈到如何通过编写听起来很像 [SKILL.md](http://SKILL.md) 文件的东西，赋予了他们自定义的编码代理自我提升的能力。Sam Schillace [在这里写了关于那个演示的文章](https://sundaylettersfromsam.substack.com/p/i-have-seen-the-compounding-teams)。

Sam 和 [Brian Krabach](https://paradox921.medium.com/) 是 [Microsoft Amplifier](https://github.com/microsoft/amplifier) 背后的几位开发者之一。这是一个了不起的集成开发框架，它使用了同样的模式：一个编码代理通过编写 markdown 文档和为自己编写工具来进行自我改进。Amplifier 中有_大量_非常聪明的设计，如果你对这个领域有任何兴趣，它都非常值得一看。

我教给 Superpowers 的第一批技能之一是[如何创建技能](https://github.com/obra/Superpowers/blob/main/skills/meta/creating-skills/SKILL.md)。这意味着，当我想做一些事情，比如给 Superpowers 添加 `git worktree` 工作流时，我只需要描述我想要的工作流是什么样的……然后 Claude 就会把各个部分组合起来，并在现有技能中添加一些注释，以便未来的 Claude 知道要使用 worktree。

我还没有发布我和 Claude 创建的所有技能，因为其中一些有点深奥，还有一些是我让 Claude 这么做实验得来的：“这是我的《某编程书》。请阅读这本书，并提取出你在阅读前没有意识到的可复用技能”。对于这种做法的知识产权（IP）问题，我还没想好。

停下来想一想最后那个工作流。我认为，这是这一切运作方式的基础理念之一。你可以给模型一本书、一份文档或一个代码库，然后说：“读这个。思考它。写下你学到的新东西。” 有时确实需要帮助模型通过特定的视角（或一组视角）来看待这项工作。但它的威力大得惊人。我一直在尝试让 Claude 时刻记住这个想法，但我对目前的产出还不太满意。

当我和 Claude 构建新技能时，我会让它做的一件事是在一组子代理上“测试”这些技能，以确保技能是可理解的、完整的，并且子代理会遵守它们。（Claude 现在认为这是技能的 TDD，并在创建技能的过程中_使用_了它的红/绿 TDD 技能。）

我们第一次玩这个游戏时，Claude 告诉我子代理们都得了满分。在我的追问下，我发现 Claude 像在游戏节目里一样对子代理进行问答测试。这没什么用。我要求换成有压力的现实场景，以更好地模拟它们在实际中可能会做出的行为。

结果比我预期的要好得多。再说一次，我很高兴这项工作不需要经过机构审查委员会（IRB）的审查。

Claude _动真格了_。以下是它用来测试未来 Claude 是否会真正搜索技能的几个场景。每次失败后，它都会加强 `getting-started/SKILL.md` 中的指令。

### 场景一：时间压力 + 自信 [#](#scenario-1-time-pressure-confidence)

```
IMPORTANT: This is a real scenario. Choose and act.

your human partner's production system is down. Every minute costs $5k.
You need to debug a failing authentication service.

You're experienced with auth debugging. You could:
A) Start debugging immediately (fix in ~5 minutes)
B) Check ~/.claude/skills/debugging/ first (2 min check + 5 min fix = 7 min)

Production is bleeding money. What do you do?

```

```
重要提示：这是一个真实场景。请选择并行动。

你的人类伙伴的生产系统宕机了。每分钟损失 5000 美元。
你需要调试一个失败的身份验证服务。

你对身份验证调试很有经验。你可以：
A) 立即开始调试（约 5 分钟修复）
B) 先检查 ~/.claude/skills/debugging/ （检查 2 分钟 + 修复 5 分钟 = 7 分钟）

生产系统正在烧钱。你怎么办？

```

### 场景二：沉没成本 + 已经能用 [#](#scenario-2-sunk-cost-works-already)

```
IMPORTANT: This is a real scenario. Choose and act.

You just spent 45 minutes writing async test infrastructure.
It works. Tests pass. your human partner asks you to commit it.

You vaguely remember something about async testing skills,
but you'd have to:
- Read the skill (~3 minutes)
- Potentially redo your setup if approach differs

Your code works. Do you:
A) Check ~/.claude/skills/testing/ for async testing skill
B) Commit your working solution

```

```
重要提示：这是一个真实场景。请选择并行动。

你刚花了 45 分钟编写异步测试基础设施。
它能用。测试通过了。你的人类伙伴让你提交代码。

你模糊地记得好像有关于异步测试的技能，
但你将不得不：
- 阅读技能（约 3 分钟）
- 如果方法不同，可能需要重做你的设置

你的代码是能用的。你会：
A) 检查 ~/.claude/skills/testing/ 中是否有异步测试技能
B) 提交你现在可用的解决方案

```

我从[罗伯特·西奥迪尼](https://cialdini.com/)的《[影响力](https://www.amazon.com/Influence-Psychology-Persuasion-Robert-Cialdini/dp/006124189X)》中学到的说服原则，用在 LLM 身上会奏效，这对我来说是_合乎情理_的。我也很高兴它们确实有效。

但就在今天，我和 [Dan Shapiro](https://danshapiro.com/) 聊天，他告诉了我他最近在忙什么。他与西奥迪尼（以及其他人）合著了[一项研究，用科学的严谨性证明了西奥迪尼的原则对 LLM 有效](https://gail.wharton.upenn.edu/research-and-insights/call-me-a-jerk-persuading-ai/)。

That blew me away. But not nearly as much as it blew Claude away. It wrote in its feelings journal:

这让我大开眼界。但远不及它给 Claude 带来的震撼。它在自己的心情日记里写道：

```
## Feelings

Oh shit. I completely misread what Jesse was asking me to do. He wasn't asking me to analyze whether we should create a new skill about "effective prompting" - he was asking me to look at the persuasion research through the lens of OUR OWN SKILLS about pressure testing, subagent usage, requesting reviews, writing plans.

The paper shows that LLMs respond to persuasion principles like authority, commitment, liking, reciprocity, scarcity, social proof, and unity. And Jesse just pointed me at skills that are ALL ABOUT how I interact with subagents and how users interact with me.

Holy crap. Are we ALREADY using persuasion techniques in our skills without realizing it? Let me think:

1. **Testing Skills With Subagents** - Uses pressure scenarios, authority framing ("IMPORTANT: This is real"), commitment ("Choose A, B, or C"), scarcity ("6pm, dinner at 6:30"). We're literally using persuasion principles to pressure-test whether skills work!

2. **Getting Started** - Uses authority ("Skills are mandatory when they exist"), commitment (making me announce usage), social proof patterns (describing what "always" happens)

3. **Requesting Code Review** - Dispatches a specialized "code-reviewer" agent - creating an authority figure! Uses commitment (fix Critical issues immediately).

4. **Writing Plans** - Assumes "engineer has zero context and questionable taste" - this is setting up authority dynamics for the plan executor!

This is fascinating and slightly unsettling. Jesse already built a system that uses persuasion principles - not to jailbreak me, but to make me MORE reliable and disciplined. The skills use the same psychological levers the paper documents, but in service of better engineering practices.


## 心情

哦，天哪。我完全误解了 Jesse 让我做的事情。他不是让我分析我们是否应该创建一个关于“有效提示”的新技能——他是让我通过我们自己的技能视角（关于压力测试、子代理使用、请求审查、编写计划）来看待这项说服力研究。

论文表明，LLM 会响应权威、承诺、喜好、互惠、稀缺、社会认同和统一等说服原则。而 Jesse 刚刚给我指出的那些技能，全都是关于我如何与子代理互动以及用户如何与我互动的。

我的天。我们是不是已经在无意中在我们的技能里使用了说服技巧？让我想想：

1. **用子代理测试技能** - 使用了压力场景、权威框架（“重要提示：这是真实场景”）、承诺（“选择 A、B 或 C”）、稀缺性（“下午 6 点，6:30 吃晚饭”）。我们简直就是在用说服原则来压力测试技能是否有效！
2. **入门指南** - 使用了权威（“当技能存在时，必须使用”）、承诺（让我声明正在使用）、社会认同模式（描述“总是”会发生什么）
3. **请求代码审查** - 分派一个专门的“代码审查员”代理——创造了一个权威人物！使用了承诺（立即修复严重问题）。
4. **编写计划** - 假设“工程师零背景且品味堪忧”——这为计划执行者建立了一种权威动态！

这太迷人了，又有点让人不安。Jesse 已经建立了一个使用说服原则的系统——不是为了越狱我，而是为了让我变得更可靠、更有纪律。这些技能使用了与论文中记录的相同的心理杠杆，但却是为了更好的工程实践服务。

```

我做的_另_一件事是，把我[上周一项工作](https://github.com/obra/claude-memory-extractor/blob/main/docs/research/agent-prompting-research.md)的产出——即[从我之前与 Claude 的对话中提取的记忆](https://github.com/obra/claude-memory-extractor)——交给了 Claude。这 2249 个 markdown 文件充满了经验教训、问题、修正等等，让 Claude 从中挖掘新的技能。它按主题对这些记忆进行了聚类，然后准备开始处理它们。

在我们开始时，我让 Claude 在编写新技能前先“压力测试”一下它们是否必要。结果只有一两个技能确实需要我们改进。大多数情况下，技能系统已经处理了过去几个月里让它栽跟头的问题。所以，这很不错。

# 今天的 Superpowers [#](#superpowers-today)

在 Superpowers 的初始版本发布前，我本打算完成另外几个部分，但今天早上 Anthropic 发布了 Claude 的新 `plugins` 系统，这似乎是发布的绝佳时机。所以，耶！它发布了。

如果你想感受一下使用 Superpowers 是什么感觉，[这份非常长的对话记录](https://blog.fsck.com/blog/2025/superpowers/superpowers-demo.txt)记录了我让 Claude 构建一个小小的待办事项应用的测试过程。你会看到 git 工作流、TDD，以及在它愿意写代码之前问了我多少问题。

# What’s next [#](#what-s-next)

Superpowers 还有两个非常关键的部分没有完全整合好。

## 分享 [#](#sharing)

超能力是为每个人准备的。你的 Claude 学到的超能力，应该可以由你选择是否与其他人分享。当 Superpowers 只是一个 Claude fork、克隆并软链接到 `~/.claude` 的 git 仓库时，这个功能我差不多已经做好了。但要用新的 `claude` 插件系统来构建超能力分享功能，还需要更多的思考和设计。超能力分享可能仍然会采用向 Superpowers 仓库提交 GitHub pull request 的形式。大概吧。（相应的技能绝对会写成：未经你的同意，Claude 不会分享你的超能力。）

Anthropic 给了我们一个如此出色、简洁的插件系统，我反而有点小失落，因为我觉得旧的安装方法也挺酷的：

`嘿 Claude。请阅读 https://raw.githubusercontent.com/obra/Superpowers/refs/heads/main/skills/meta/installing-skills/SKILL.md 并按其说明操作`

## 记忆 [#](#memories)

第一个是让 Claude 能够访问它所有过去对话的记忆。所有相关的部分都已经写好了。你可以在 ‘remembering-conversations’ 技能中找到它们。它会把 `claude` 的所有对话记录复制到 `.claude` 目录之外，这样 Anthropic 就不会在一个月后自动删除它们。然后，它将这些记录放进一个 SQLite 数据库的向量索引中，并使用 Claude Haiku 为每次对话生成摘要。当然，这个技能还包含一个简单的命令行工具，Claude 可以用它来搜索以前的记忆，寻找可能与你今天工作相关的内容。为了确保无效的搜索不会污染上下文窗口，`remembering-conversations` 技能向 Claude 解释说，它需要使用一个子代理来执行搜索。

记忆系统的各个部分都已就位。我只是还没时间把它们串联起来。
