# 38 - 《把小事做大》

另一种找事做的思路。

前面的问答中有人问「该做的事都有人做了，我可以做点啥」；前几年我做规划时也有苦苦追寻那些一鸣惊人的可以大刀阔斧去干的「大」事，做调研，挖痛点，跟公司和团队的趋势等等。

大事应该做，但不常有，或者早就被人抢了。这时可以换个思路，把小事做大。比如我们做框架，对于开发者来说，大事不一定有用，小事可能更有意义。比如实现一个可视化编程，开发者觉得不好用；而修复 git commit 提交速度的问题，开发者就很开心。

小事做大可以是质的也可以是量的。前者指做极致，把一件事做到能力范围内最好；后者指多做，重复做，有规划有周期性地做，比如每天或每周或每月做，一年后回过头看，也很有意义，不管述职还是绩效里都是亮点一个。

某 P9 大佬近期的签名是「每天解决一个问题」。内网前端框架 Bigfish 今年也有个计划是「每周解决一个问题」，Bigfish 框架已有 3 年+，中间小问题其实不少，虽影响开发者体验，但优先级相对低，一直被往后排，希望借此方式，让这些问题得到解决。
