# 412 - 《研发效率、Low Code、Pro Code》

> 这是一篇发在内网的文章，整理下贴到这。背景是有同学做了 Low Code 和 Pro Code 的对照实验，看完之后有些想法，就整理了下。

**先说对照实验。**

1、**和研发效率正相关的不是代码量，而是解决问题的时间。** Pro Code 追求不是代码量，是简单、DRY 之类的原则，以及用更简单的方案更少的代码去解决问题。为啥是更少的代码？因为这通常意味着更少的维护成本。同时随着项目的迭代和抽象，Pro Code 解相同问题所需的代码量会越来越少。

2、**问题不是一次性的，还需考虑后续迭代成本。** 对照实验里只做了「实现需求」的对照，没有做可维护性的对照。Low Code 用地不多，不清楚这部分的对比数据会如何发展。盲猜 Pro Code 同学在做需求时，除了研发速度，还有其他考虑，比如可维护性，为了可维护性，前期可能会做不少设计、抽象、选择和工程化，这些成本是一次性的。。如果对照实验的目的是为了对比二者的研发速度，不妨直接告诉他们，可能结果会不太一样。

3、**解问题除了看快慢，还可以看看过程和结果。** 对照实验不确定有没有做结果验证，翻看了其中几个 Pro Code 的代码，有的结果粗糙，有的细致，他们所需要的时间肯定也是不同的。此外，也还有性能、可扩展性、研发体验、用户体验等值得考虑。

**回到 Pro Code，感觉有很多提升空间。**

都这么多年了，我们完成如下一个 CURD 加详情页（不算联调）居然还需要 3 天，这个流程上肯定是有优化空间的。翻了下代码，开发同学是用 antd 组件裸写的，如果换成 ProTable，或者用上 TechUI Studio，感觉一天顶天了。CURD 是举例，也适用于其他场景。

**所以，我有几件想做的事。**

1、**SOP。** 从 PRD、系分、数据（OneAPI）、流程、组件库选择、CSS、数据流、请求方案、燕鸥、调试、埋点、错误处理等，我们有各种方案，但没有 SOP，并且文档也散落在各处。所以开发者拿到 CURD 需求时，还需要各种思考、选择、搜索和问人。如果有个 SOP 文档告诉开发者从 1 到 10 做 10 步就可以完成需求，我觉得是可以提效的。

2、**培训。** 对照实验的 W1 和 S4，W1 应该是培训过主职做 Low Code 的，而 Pro Code 本身目前没有系统的培训课程，文档也是零散的，Bigfish 的视频还是上一个版本的。对于新人比较多的场景，需要有从 Bigfish 和 Pro Code 角度的培训。教怎么做的快和怎么做的好。

3、**高质量标准化智能化的区块。** 区块是可以落到用户项目的代码片段，是组件之上的抽象。这是好多年前做 Umi UI 时提过的概念，当时没做成我觉得原因是区块质量不高。去年 shadcn/ui 大火，火的原因我觉得除了他本身的高质以外，他并不是和其他组件库一样，以 npm 包的方式提供，而是通过 CLI 工具，直接复制源码到用户项目中，以及安装必要的依赖，这给了用户对代码更大的掌控权。所以，我的思路是，为标准流程提供高质量区块，然后再结合 AI 支持内容微调。
