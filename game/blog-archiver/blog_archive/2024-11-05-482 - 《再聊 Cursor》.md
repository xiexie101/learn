# 482 - 《再聊 Cursor》

> 这段时间编码都用 Cursor，并且会有意识地尽量用 AI 辅助编程，不仅仅是 Tab 提示，更多是 Composer 和 Edit 的功能。感觉效率提升不少，就再补充一些信息吧。

1、免费用。

Cursor 目前是可以免费刷会员无限用的。因为他有免费试用 14 天的机制，同时注册邮箱也没有做 alias 的限制。所以如果你有 alias 功能的邮箱，比如 gmail，刷会员就很简单，注册的时候用 Alias 就好。比如我的邮箱是 [sorrycc@gmail.com](mailto:sorrycc@gmail.com)，过期了用 [sorrycc+cursor1028@gmail.com](mailto:sorrycc+cursor1028@gmail.com) 再注册一个，然后在 Cursor 编辑器里换个账号就好。

2、文字游戏。

编程也成了文字游戏。你只要能描述需求，就能实现功能。我现在也养成了这样的习惯，先详细描述需求，然后交给 Cursor Composer，他搞不定我再自己上。当然，这也需要有编程基础，不然不知道 ai 写的如何，以及在他出错时不知道如何解。

3、效率。

Cursor 能提效多少？这和很多问题的答案一样，「By Scene」。对不同人、不同场景、不同需求所带来的效率提升应该都是不同的，不能一概而论。[Santiago 说](https://x.com/svpino/status/1847608683410444571) AI 能帮他写 5% 的代码，然后节约 20% - 30% 的时间；[马天翼说](https://x.com/fkysly/status/1852981759073526146) Cursor 帮他写完 1K 行的代码，他基本没写。我的感受是，简单需求能提升 50% 甚至更多，复杂需求能提升 20% 左右。

4、双刃剑。

也有一些对 Cursor 反对的声音，比如，1）[用 Cursor 之后对代码的熟悉程度降低了](https://m.okjike.com/originalPosts/670506d6e472559742bce11c)，2）Cursor 产出的代码质量可能没那么高。我对 Cursor 是持积极观点的，但是无脑用 Cursor 而不看代码或者看不懂代码，确实后期容易崩，AI 不能保证一直正确。另外有个思路是，让 AI 生成测试用例，我们确保用例是过的，然后具体的实现当成是黑盒，可能也是一种使用他的方法。

![](https://tcc.sorrycc.com/p/01605b7e-3af2-4325-97fb-641af588b367.png)

5、Cursor x TNF。

我在写 TNF 时，一些简单任务就直接上 Cursor Composer，比如加个 CI，加个 e2e 测试脚本，加基础的 Dev 功能，加 prettier 和 husky 能力等等，这些通用的任务，自己做也是看文档或者从其他仓库里复制，重复劳动。只要描述清楚需求，AI 做的会比人更好。

当然也有遇到 TNF 搞错的任务，比如我让他用 serve 实现 server，但 serve 并没有提供 createServer 的编程能力，他认为我是对的，直接用 serve 的 createServer，但实际上没有。后面我查文档后，让他改换 serve-handler 后搞定。

有了 Cursor 之后，能描述清楚需求和思路的能力会比较关键。另外，Cursor 只是工具。如果自己的思路是模糊的，或者不能评判 Cursor 出的代码质量的好坏，用 Cursor 可能会让烂代码越堆越多，最后成为屎山。
