## 2025-01-30

*   **[新的一年，如何保持精力充沛？](https://mp.weixin.qq.com/s/qNV-30DLqsfnEAWrpgjejg)** 这篇文章从运动、饮食和休息三个方面讨论了如何在新的一年保持精力充沛。1) 运动方面，建议关注心肺功能而非单纯练肌肉，通过监测心率来控制运动强度，35 岁的人中高强度运动心率应控制在 160 次/分钟左右。2) 饮食方面，推荐每餐遵循 211 饮食法：2 拳头蔬菜、1 拳头高蛋白、1 拳头主食，避免过多摄入高糖高油食物。3) 休息方面，建议远离电子产品蓝光，保持规律作息，关注深度睡眠质量。文章强调精力管理是一生的课题，虽然改变生活习惯很难，但为了能抓住机会享受美好生活，这些改变都很有必要。
*   **[Intro to Wasm in Deno](https://deno.com/blog/intro-to-wasm)** 这篇文章介绍了如何在 Deno 2.1 中使用 WebAssembly (Wasm)。Wasm 是一种低级的二进制格式，能让 C、C++ 和 Rust 等语言编译后在浏览器中以接近原生的速度运行。文章通过两个例子展示了 Wasm 的使用：1) 用 WebAssembly 文本格式 (wat) 写了个简单的加法函数，编译成 wasm 后在 Deno 中调用；2) 用 wasmbuild 工具创建了一个 Rust 项目，包含加法函数和打招呼的功能，然后编译成 wasm 供 JavaScript 调用。整体来说，Deno 2.1 让导入 Wasm 模块变得特别简单，就跟导入普通的 JavaScript 模块一样方便。文中还提到了 Wasm 在 Google Earth 和 Photoshop 等项目中的应用，以及它在金融和医疗平台中的安全性优势。

## 2025-01-28

*   **[How to improve your WFH lighting to reduce eye strain • Russell Baylis - Designer](https://rustle.ca/posts/articles/work-from-home-lighting)** 这篇文章讲述了如何改善居家办公 (WFH) 的照明，以减轻眼睛疲劳。文章作者分享了自己调整 WFH 照明环境的经验，强调了以下几点：1) 均匀、漫射的照明环境对眼睛最好。2) 光线太强和太弱都存在问题。3) 尽可能利用自然光。4) 人工照明的质量很重要。5) 为摄像头提供的最佳照明，不一定是符合人体工程学的最佳照明。6) 即使是完美的照明环境也会使人疲劳，需要休息。文章指出，电脑屏幕的闪烁、高对比度照明以及长时间固定焦距都会导致眼睛疲劳。作者的改进方案包括：使用滤光窗帘和木百叶窗来控制和漫射自然光，在房间角落增加落地灯以创造均匀的照明，使用可调光无闪烁的背光灯和显示器灯条来匹配显示器的亮度。作者还推荐使用高质量的 LED 灯泡。建议在考虑人体工程学时，除了照明，还应注意休息，保持良好工作习惯，如 20/20/20 法则，以及调整姿势和工作环境，同时注意房间湿度。

*   **[JavaScript Temporal is coming | MDN Blog](https://developer.mozilla.org/en-US/blog/javascript-temporal-is-coming/)** 这篇文章介绍了 JavaScript 新的 `Temporal` 对象，旨在取代存在缺陷的 `Date` 对象。`Date` 对象仅支持本地时间和 UTC，缺乏时区支持，解析行为不可靠且可变，易产生 bug。`Temporal` 提供了更可靠和可预测的日期时间管理，支持时区和日历表示，内置转换、比较和计算方法。1) 核心概念包括 `Duration`（时间差），`Instant`（时间戳），`ZonedDateTime` （带时区的日期时间），`PlainDateTime`（不带时区的日期时间）等等。2) 它通过 `Temporal.Now` 获取当前时间，并能处理不同日历系统（如中国农历），以及 Unix 时间戳。`Temporal` 提升了时间计算的精确度和效率，并提供了 `compare()` 方法，方便对时间进行排序。目前，`Temporal` 正在浏览器的实验版本中逐步实现，Firefox 的实现较为成熟。开发者可以通过 `js-temporal/polyfill` 在 TC39 文档页面体验 `Temporal` 的功能。这篇文章也感谢了为此做出贡献的人。 [译：JavaScript Temporal 即将到来](https://sorrycc.com/javascript-temporal-is-coming)
*   **[The Most Mario Colors](https://lmnt.me/blog/the-most-mario-colors.html)** 这篇文章探讨了 马力欧 (Mario) 游戏 logo 中 “Mario” 字样的颜色搭配。作者分析了 40 款采用多边形风格且 Mario 字样为多色的游戏，并排除了重制版和彩虹色排序。分析结果显示：1) 红色是 M 最常用的颜色，绿色是 A 最常用的颜色，黄色是 R 最常用的颜色，蓝色是 I 最常用的颜色，绿色是 O 最常用的颜色。2) 整体而言，绿色是最常用的颜色，总共出现了 55 次。3) 常见的颜色组合是 红-绿-黄-蓝，这种组合的顺序经常出现。文章还观察到，最近的游戏 logo 越来越倾向于使用 红-绿-黄-蓝-绿 或 红-绿-黄-蓝-粉 的颜色组合，就连《超级 马力欧 兄弟 大电影》的 logo 也采用了 红-绿-黄-蓝-绿 的配色，最后作者幽默地承认这篇分析可能没什么实际用处。

*   **[AI Coding 编辑器没有那么神秘 - How Cline works](https://www.nazha.co/posts/how-cline-works)** 这篇文章主要介绍了 AI 代码编辑器 Cline 的工作原理。Cline 的核心是利用大语言模型 (LLM) 的指令遵循能力，通过系统提示词、用户输入和项目环境信息来引导 LLM 完成编程任务。它的工作流程大致如下：1) Cline 将系统提示词（包括工具使用规则）、用户输入的任务指令，以及项目环境信息（例如文件列表、打开的标签页等）打包发送给 LLM；2) LLM 根据这些信息生成思考步骤和操作指令，例如执行命令、读写文件等；3) Cline 解析 LLM 返回的指令，执行相应的操作；4) Cline 将操作结果和之前的输入再次发送给 LLM，并重复这个过程，直到任务完成。 Cline 使用 XML 格式来定义工具的使用方式，并支持通过 MCP 服务器扩展功能。此外，Cline 的一个特点是直接将所有信息发送给 LLM，可能会导致 Token 使用量较高，并容易触发 LLM 的上下文窗口限制，它会暴力截断处理。文章推测其他 AI 代码编辑器的实现方式可能与之类似，例如通过 `.cursorrules` 文件注入系统提示词，而不是对代码库进行嵌入。 \[\[AI Coding 编辑器没有那么神秘 - How Cline works-1737732000126\]\]
*   **[Initial load performance for React developers: investigative deep dive](https://www.developerway.com/posts/initial-load-performance)** 这篇文章深入探讨了 React 应用的初始加载性能，主要关注首字节时间 (TTFB) 、首次内容绘制 (FCP) 和最大内容绘制 (LCP) 这几个核心指标。文章首先介绍了这些指标，以及它们如何影响用户体验。1) TTFB 是浏览器发送请求到接收到首字节的时间；2) FCP 是用户首次看到屏幕上有内容的时间，理想值低于 1.8 秒；3) LCP 则衡量页面主要内容何时加载完成，理想值低于 2.5 秒。接着，文章详细讲解了如何使用 Chrome 开发者工具的性能面板和 Lighthouse 来测量这些指标。性能面板可以模拟不同的网络环境，例如慢速服务器、低带宽和高延迟，从而帮助开发者识别性能瓶颈。文章还通过一个简单的 React 项目来演示这些概念并做实验。文章强调了缓存控制的重要性，特别是 `Cache-Control` 标头。1) 通过合理的 `max-age` 设置，浏览器可以缓存静态资源，减少重复加载的时间。2) 现代打包工具可以通过哈希文件名来自动实现资源版本控制，有效利用浏览器缓存。文章最后提醒开发者，即使使用了现代工具，仍然需要检查 CDN 和托管服务商的缓存配置。 总而言之，文章旨在帮助 React 开发者理解初始加载性能的关键因素，并掌握优化工具和策略。 [译：深度调研 React 开发者的初始加载性](https://sorrycc.com/initial-load-performance)

*   **[The Seven-Action Documentation model - passo.uno](https://passo.uno/seven-action-model/)** 这篇文章介绍了一个名为“七行动文档模型”的文档构建方法，它强调从用户需求而非内容类型出发来组织技术文档。文章认为，现有的文档框架和工具过于关注技术写作者的行动，而忽略了文档消费者的需求。作者提出，文档应该被视为一个产品，其目标是满足用户的特定操作需求。这个模型包含七个关键的用户行动：1) 评估 (Appraise)，帮助用户了解产品的主要特点；2) 理解 (Understand)，解释产品概念；3) 探索 (Explore)，引导用户尝试新功能；4) 实践 (Practice)，教授用户如何操作；5) 记忆 (Remember)，提供参考信息；6) 开发 (Develop)，指导用户进行扩展和集成；7) 故障排除 (Troubleshoot)，帮助用户解决问题。作者认为，这七个行动涵盖了用户与文档交互的主要模式，并且可以在不同的文档层级和范围中得到满足。这个模型并不限制内容类型与行动的对应关系，鼓励文档同时满足多种用户需求。最后，作者展示了一个基于这个模型的示例文档结构，指出该模型可以帮助技术写作者创建更有针对性和实用性的文档，并可以用于文档的度量和目标设定。
