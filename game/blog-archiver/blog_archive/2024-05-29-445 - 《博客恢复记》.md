# 445 - 《博客恢复记》

> 还是花了点时间的，记录下。

如 [353 - 《VPS、宝塔、Blog》](https://sorrycc.com/vps-baota-blog) 所言，我的博客是基于宝塔 + WordPress 搭的，但是两周前发现宝塔面板登录不了，于是一通折腾，升级了宝塔面板，然后。。。彻底坏了。

![](https://res.cloudinary.com/sorrycc/image/upload/v1716948029/blog/h5n1wu3s.jpg)

本能还能跑的原则就没管他，谁知上周末数据库也挂了。

![](https://res.cloudinary.com/sorrycc/image/upload/v1716948313/blog/1w72ypz6.jpg)

好在之前有用 [WPvivid 备份插件](https://cn.wordpress.org/plugins/wpvivid-backuprestore/) 每天早上 8 点定时备份到 Google Drive，数据倒是不用担心。于是找了台机器装 1Panel，部署 WordPress，装 WPvivid 备份插件，然后恢复。先是用的 vkvm 的 2C2G 的香港机器，想着访问上应该会快一些。但是搞定后发现 CPU 基本占满，响应很慢。。一个博客怎么会这么占 CPU？想着可能是机器的问题，于是换了搬瓦工的 1C2G 的日本机器，相同的操作，结果完全没问题。

用 [https://ping.chinaz.com/sorrycc.com](https://ping.chinaz.com/sorrycc.com) 测了下，速度比之前应该快不少。

![](https://res.cloudinary.com/sorrycc/image/upload/v1716949008/blog/j3k97udx.png)

然后又遇到更新内容的问题。我之前是用 XML-RPC 的方式更新博客内容，但是跑的时候遇到以下错误，原因未知，google 和 chatgpt 了下也没找到结果。

```
error: Unknown XML-RPC tag 'LINK'
```

尝试换 WordPress 的 [Rest API](https://developer.wordpress.org/rest-api/) 更新数据，但是又遇到诡异问题，当内容里同时出现「链接」和「图片」时，会返回 403 响应，原因未知。

最终，我把内容更新的部分换成直连数据库的方式更新，算是临时解吧。
