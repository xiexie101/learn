# 621 - 《AI Coding FAQ》

> 对群里的问题做下快问快答。可以评论里接着问，新问题的答案我会补充到前面。

## 把 plan 提交到仓库有什么作用吗？分享下你的思考？

要回答这个问题，首先得明确 plan 是什么。大家对于 plan 的理解可能都不太一样，有的简单，有的复杂。我就先把它理解成 spec 驱动里面的 plan 文档。如果基于此，那我并不会把 plan 提交到仓库，而是会把 design 提交到仓库。我在 [neovate-code](https://github.com/neovateai/neovate-code/tree/master/docs/designs) 和 [neovate-code-desktop](https://github.com/neovateai/neovate-code-desktop/tree/master/docs/designs) 里都是这么做的。

![](https://pic.sorrycc.com/proxy/1770217677192-134111106.png)

为什么不提交 plan 文档？原因有几个，第一，plan 文档太复杂，这是给 ai 读的，不是给人读的，第二，我的 spec 驱动的流程大多（99%+）都不会走到 plan 这个流程，我走完 design 就会要求实施，因为大多需求都不会复杂到这个程度，复杂的需求我会人为把他拆小。

为啥要提交 design 文档？原因也有几个。第一，设计文档记录了你脑暴和思考的过程，这份文档很有价值。第二，他可以方便团队其他同学在方案实施前进行讨论和 review，也可以让 AI 后续来读。再补充个今天还遇到的场景是，我忘记了当时某个代码为啥这么写，通过 git blame 查找和回忆有点麻烦，通过关键词去找到 design 文档就很快知道了当时这么做的原因。

## 目前程序员如何编程，比如 codex，cc 都做出了 app ？

我觉得最重要的是模型，要舍得花钱或者花精力去研究，用上目前最好的模型。至于用啥工具，找个顺手的就行，只要不太拉，常用功能够就好，我 95% 以上的代码都是用 @neovate/code + opus 4.5 产出的。CLI、IDE、APP 用啥都可以，对效率的影响其实没那么大。

App 值得尝试，但不用马上追。有几个考虑，1）模式的转变，这标志着你从以 Editor 编程为主转换到了 Agent 编程为主，Agent 编程模式下，Editor 的作用大幅减少，已经很少需要人为去编码，2）异步编程模式提升产出的上限，当你能力足够时并行去做多任务编程时，App 的方式不会限制你，你可以在一个窗口下同时进行 N 个任务的编程。下图是 Neovate Code Desktop 的截图。

![](https://pic.sorrycc.com/proxy/1770218749496-359165068.png)

我说不用马上追的原因是，1）大多人 2-3 个任务并发到顶了，就目前大多 App 的功能而言，这时多开几个 Editor 和 CLI 窗口效果是一样的，2）基于 git tree 的 workspace 能力个人感觉是伪需求，可以有 n 种其他方法来解，3）remote background 也不是强需求，大多编程场景还是得在本地。

所以，可以先试用一下，不习惯就回到之前的 CLI 或 IDE，但你也可能会很喜欢。

## 关于 skills 工作流？

Skills 我觉得是趋势，而且是强需求，2026 必学。MCP 我是一个都不会用的，而 Skills 我有 N 个自研在用的。而且 Skills 很简单，name + description + prompt + 可选的 scripts 定义即可。

为啥 Skills ？1）门槛低，遇到解过的问题或者 SOP 随手就可以创建一个 Skills 复用，2）Skills 可以按需载入，大幅解了之前 MCP 占用上下文的问题，但这里说大幅而不是彻底解是因为 Skills 还是会通过 name + description 小幅占用一点上下文的，所以不要全局安 n 个 Skills！！！3）Skills 可分享、可组合、可发现、可被主动调用，可组合是可以先调 a skill 再调 b skill 也可以在 a skill 里调 b skill，可发现是 ai 可以基于 name + description 决定用哪个 skill，可被主动调用是大多工具都有 skill as slash command 的功能，可以把 skill 作为函数调用，总之，Skills 很灵活。

我个人是把自己的脚本和工作流都用 skills + scripts 重构了一遍，可以随时调用和组合。Scripts 可以用 python 或 bun 组织，我用 bun 偏多。

## 用save保存的文档 经常开发到后面就偏离了。然后后面检索会检索到md文档。很容易出现幻觉

所以，方案变更后，我通常会要求 `Update Design Doc` 。而且感觉正常来说，AI 不会主动检测这些 md 文档。

## 给AI做的spec通常非常细，但是人很难全心跟完整个计划到执行的流程，那么这里有没有什么好的实践，或者平衡的方法

不要走全流程的 spec ！！！没必要，又臭又长，浪费时间，也浪费 token 。

分享我个人 3 个不同 level 的 spec 驱动的方式。

1）最简单，覆盖我 60% 的场景。就是要求执行的时候多加下面一句话。这时 AI 就不会直接改，而是会先 analyze 和找 root cause，和你确认方案后再开始改，这用于应对大部分的简单需求，就足够了。

```
Don't implement or fix directly, analyze related files first if needed, tell me what changes are needed and ask questions if unclear.
```

2）脑暴模式，覆盖我 30% 的场景。当方案有点复杂，尤其是需求本身我自己描述不太明确时，我会用这个。Neovate Code 有内置这个功能，通过两次 shift+tab 切出来，具体不展开了。这个来源自 [superpower](https://github.com/obra/superpowers/tree/main/skills/brainstorming)，大家也可以直接用他的 SKILL 。好处就是一步步和你明确需求，确定方案，然后产出 design doc，这个出来的方案，质量会很高。

3）Plan 模式，claude code 新的 plan 模式，Neovate Code 也学习了一版。这个 plan 会先对项目做 analyze，然后也优化了体验，对于相比脑暴来说更复杂点的需求值得一试。但其实大多用脑暴模式就够了。

## Neovate 相较于编辑器和插件这种可视化代码的优劣势（想听cc老师吹一下，因为我一直在努力给央企领导推这个，领导总是把目光投向opencode那种东西上

万一领导是对的呢。。

其实是个人习惯问题，有人喜欢用 CLI，也有人喜欢用 GUI。OpenCode 也有 CLI 和 GUI 的形态。Neovate 主要是 CLI，也在做 Desktop（GUI）的形态。

如果从开源热门的角度来选，Opencode 目前来看更占优势，而 Neovate Code 的优势是二开会更友好，包括 CLI 和 Desktop 的二开，就看你们有没有这类的需求了。

Agent 的能力上其实相差都不大，大家抄来抄去，能力基本上都是对齐的。

## neovate 可以用 antigravity 的api吗

可以。

1）拥有 antigravity 的账号  
2）下载 [https://github.com/lbjlaq/Antigravity-Manager](https://github.com/lbjlaq/Antigravity-Manager) 切账号登录然后开 API 反代  
3）修改 `~/.neovate/config.json`

```
{
  "provider": {
    "antigravity": {
      "options": {
        "baseURL": "http://127.0.0.1:8045/v1",
        "apiKey": "你的 KEY"
      },
      "models": {
        "claude-opus-4-5": {},
        "claude-4-5-sonnet": {},
        "gemini-3-flash": {},
        "gemini-2.5-flash": {}
      },
      "apiFormat": "anthropic"
    },
  }
}
```

4）`neovate /model` 切 antigravity 下的 claude-opus-4-5 。

至于为啥 Neovate Code 把之前的 antigravity Provider 删了，原因是，1）Antigravity 只通过 oauth 的方式不稳，个人用下来发现用 Antigravity Manager 是最稳的，以及新账号还需要在 Antigravity 登录后再用啥的，2）找到更好的方式后 Antigravity Provider 的维护优先级变低，目前不可用，不如把他删了。
