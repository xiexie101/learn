# 23 - 《晋升或述职建议》

上月做了一天晋升评委，回来后整理给团队外包同学的建议。同样适用于 P5-P6 和 P6-P7。给有述职任务的同学，或者有晋升想法的，也可以对照下做计划。下文的 Bigfish 是基于 Umi 的内网框架。

1、硬指标：代码提交次数 + 某文档平台创作指数。面试官会看，数据好有明显加分；数据差可能会成为卡点。所以 1）养成小 PR 的习惯，不要经常很大了一次性提交，commit 拆分也要合理一些，一个 commit 做一件事 2）注意总结和分享，比如之前实现了一遍 low import，可以写写「如何入门 babel 插件」或者「low import 实现心得」，既是一次整理总结，对于未来晋升也有帮助

2、清楚业务方的目标是啥。不能只埋头干，比如我们的业务是 Bigfish，不仅要知道现在在做的事，还要知道 Bigfish 的规划

3、利用技术达成业务目标。Bigfish 做地事情本身就包含大量技术，但要注意的是学会找点，在做的过程中找到可以提升 Bigfish 的点，会是面试中的亮点，比如做了一段时间答疑后想到一个答疑的改进方法，或是重构了强约束实现，都算

4、数据支撑。前一点里提升的地方是否有数据支撑，所以做的时候就需要掌握一些数据，比如做答疑改进，做之前如何，做之后如何。

5、你的角色。做的事情里，你的角色是什么，发挥了什么作用。这是面试时最常见问题之一，可以加一些语言修饰，但需要大部分是自己的。

6、抽象和复用。做完一个方案只能用在自己项目是本职要求，做完后做抽象提取，可以被其他项目复用才会被面试官认可。此外如果有有数据支撑，被多少项目使用了就更好了。面试时的常见问题。

7、意识。经常问自己「xx 的最大痛点是啥？」，xx 可以是 Bigfish，或者 Bigfish 的某件子任务或某个子领域。面试时常见问题。

8、个人规划。清楚自己想做成什么样，包含个人成长、技术、团队（如有）等，加分项。

9、逻辑性。讲的时候要注意下。因为 A 所以做 B，注意这个逻辑的正确性和推导关系，比如今天一个案例，因为构建卡，所以要用 qiankun 拆子应用，这个前后就没有强关联。
