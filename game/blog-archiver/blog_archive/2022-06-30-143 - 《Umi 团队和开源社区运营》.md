# 143 - 《Umi 团队和开源社区运营》

1、等级制。其实看下各个开源库的 [CONTRIBUTING.md](http://CONTRIBUTING.md)，很多都有分等级制，用以激励大家更积极地参与社区建设。Umi 贡献者分 Contributor、Member 和 Core Member。一个有意义的 PR 即可成为 Contributor 并入群交流；Member 是 10 PR 的门槛；Core Member 随缘，可能需要杰出贡献并长期参与。

2、周会。会邀请 Contributor 群里的所有同学参加，非实线成员不强制，不用请假，爱来就来，哈哈。时间是每周四 11 点到 12 点，控制在 1 个小时，速战速决。议题是固定的，固定议题可以让周会机制更长久，1）答疑问题 Review，2）一周一问题计划轮流分享，3）随缘分享，4）下周迭代梳理。

3、实线团队和虚线团队。实线团队是维护的有力保障。之前 Umi 实线只有 1 + 0.5 或 1 + 0.5x2，其他都是虚线。在经过几次组织架构的调整之后，虚线团队基本就散了，也不是大家不愿意参与，而是和 KPI 毫无关系又没有老板要求的事，纯用爱发电是很难坚持的。所以，后来 Umi 开始组建实线团队。目前有 3 正式，1 在途正式，1 在途实习，2 外包组成。虚线有来自其他业务线的同学，以及来自五湖四海各大中小厂的朋友们。

4、反馈。内部通过钉钉群、内部任务管理系统；外部通过微信群、钉钉群、issue。搞了个子域名 [fb.umijs.org](http://fb.umijs.org) 指向腾讯文档。由于公司电脑不能安微信客户端，微信群主要以交流和互助为主，钉钉群会有 Umi Contributor 参与答疑，但要求「提供最简复现」，用以节省大家的时间。通过整理最简复现，其实很多问题的解会自己浮出水面。

5、通知。比如版本发布，新功能预告等。目前是通过群公告、我的朋友圈、知乎专栏、我的公众号发布文章的形式通知，但感觉不太好，触达率比较随缘。后续会改成官方带 RSS 的 Blog 和 Newsletter 的形式，新版 [umijs.org](http://umijs.org) 右下角新增了 Newsletter 的订阅入口，其服务基于 twitter 的 [getrevue.co](http://getrevue.co)。

6、任务流。Umi 由于有来自「五湖四海」的社区朋友一起参与研发，一直在探索大家一起协作的方式。先是尝试了钉钉的待办，但发现非阿里钉的 Mac 版本无此功能。然后最近开始换 Github 的 Projects 看板，用「Requments ➡ Problem Defined ➡ RFC ➡ In Progress ➡ DONE」的状态流，定义清楚问题和讨论清楚方案都很重要，这么设置也是希望大家不要做「问题模糊、解决方案有争议」的任务，「Think Before acting」。

7、发布周期。正常情况下，由实现正式员工轮值发布，时间是每周四下午。特殊情况比如遇到紧急 BUG 随时可发。同时，日常有需要会随时发 canary tag 的版本，用于验证问题或者让急用的同学临时用用。
