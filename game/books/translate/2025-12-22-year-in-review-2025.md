# 2025 年度回顾

**Mario Zechner** • 开发者 • 教练 • 演讲者
**2025-12-22**

距离我上一次写年度回顾已经过去一年半了（那次实际上涵盖了两年）。随着 2025 年接近尾声，我想以此总结一下这一年我在家庭和商业之外的生活。像往常一样，这是技术、黑客行动主义和各种胡扯的混合体。我想把这一切写下来，希望能简洁一些。对我来说，这是为了留下记忆；对你来说，希望能从中找到一些有趣或有用的片段。

如果你看一眼顶部的标题图片，你会发现二月和三月有一段空白。原因是：我变老了。我想这是我今年最有趣的观察。我的身体正在慢慢垮掉。我在医院住了一段时间，然后需要相当长的时间来恢复，这意味着除了家庭事务外，我没有精力做任何事情。这也解释了那一时期我 GitHub 贡献的空白。

好了，言归正传。让我们从我最自豪的一个项目开始。

这真是一段旅程，从基本的 LED 闪烁到自行设计印刷电路板，像疯子一样购买所有装备，并熟练掌握 SMT 元件的焊接。显然，我还必须将《毁灭战士》（Doom）移植到这个嵌入式系统中。

很高兴向大家报告，我们的小家伙至今仍每天使用它，现在拥有一个装有约 72 个卡带的宝箱，每个卡带里都有他最喜欢的节目、有声读物和音乐。

每当有东西坏了（这种情况出奇地少），孩子和我就会一起拧开设备，查看内部结构并在里面通过挖掘来修理。我希望这种早期的技术接触能激发他对事物运作原理的兴趣，就像 Game Boy 曾经激发了我的兴趣一样。

## 乌克兰援助卡 (Cards for Ukraine)

很高兴向大家报告，今年我们的 Cards for Ukraine 项目筹款突破了 **30 万欧元**（你可以阅读[起源故事](https://mariozechner.at/posts/2022-03-02-cards-for-ukraine/)）。许多小额捐款和一位匿名捐赠者的大笔捐款帮助我们实现了这一目标。

我们发出了两批援助，一批在 5 月（190 张卡），另一批在 10 月（308 张卡），这些卡片被送到了在奥地利的乌克兰家庭手中。Tanja 仍在写她的 [Substack](https://tanja.substack.com/)，并经常在文章中呼吁捐款。而我则继续尝试用愚蠢的项目来娱乐互联网，并在每篇社交媒体帖子的末尾乞求大家为该协会捐款。这仍然有点用，但显然在经历了三年的战争和经济衰退后，人们的钱包不再像以前那样慷慨了。

## heisse-preise.io

[heisse-preise.io](https://heisse-preise.io/)（[起源故事](https://mariozechner.at/posts/2023-08-01-heisse-preise/)）继续稳步运行。在 2024 年平安夜，SPAR 决定更改其 API，这破坏了对其产品的价格采集。我在三月份注意到了这一点，但没费心去修。直到五月我才终于腾出手来解决。因此，SPAR 产品从 2024 年 12 月 24 日到 2025 年 5 月 14 日有一段数据空白。这就是爬虫的乐趣。

多年来，我一直呼吁奥地利政府实施 BWB（联邦竞争管理局）2023 年的建议：将食品价格数据抓取合法化，并要求零售商公开 EAN 代码。杂货商已经通过 API 公开了数据，BWB 发现他们自己也在使用这些 API 进行“系统性价格匹配”。消费者应该拥有同样的访问权限。但 Kocher 部长在实施任何措施之前就去了奥地利国家银行（OeNB），而新政府同样无能或不愿解决这一个法律问题。这不仅能让价格比较网站立足于良好的法律基础之上，还能为科学调查开放数据。没有法律保障，没有大学会碰我的数据。

有趣的是，Cory Doctorow 在接受 The Verge 采访时提到了我，称我为“那个抓取了两大杂货商所有网络价格的奥地利小孩”。我都 41 岁了。

关于 heisse-preise.io 最酷的事情仍然是我妈妈用它来对价格发火。

> “你好。每天我的血压都会因为这些难以置信的事情升高……现在 1 个鸡蛋要 40 分。哦等等，里面有 15 个。算是 26 分一个。但随着价格上涨，它每天都在涨。”

## 搞搞 AI (Messing with AI)

自 2023 年初 ChatGPT 发布后不久，我就一直在玩大型语言模型。参见我去年的年度回顾，了解 AMS Berufsinfomat、生成式 AI 工作坊、Leitkulturomat 和 KärntenGPT 等项目。

在过去的三年里发生了很多事情。我们基本上已经从在网站上询问 ChatGPT 或 Claude 一些功能，转变为在我们的机器上以 YOLO 模式运行半自主代理（Agents）。从今年第二季度开始，我完全接受了这种工作流程，并在博客上记录了这些：[Prompts are Code](https://mariozechner.at/posts/2025-06-18-prompts-are-code/)、[MCP vs CLI](https://mariozechner.at/posts/2025-07-02-mcp-vs-cli/) 和 [What if you don't need MCP](https://mariozechner.at/posts/2025-07-16-what-if-you-dont-need-mcp/)。早期我还录制了一个 1.5 小时的视频，详细介绍了我是如何与编码代理（Coding Agents）一起工作的。回顾过去，我的工作流程已经发生了显著变化。我最终会写一篇单独的博客文章。可以说，它还远非完美，但感觉更加流畅、稳定和高效。重点在于**感觉**。因为这是我们拥有的全部——感觉（Vibes）。

Armin 在他最近的博客文章中将编码代理用户分为两个阵营：一类是将代理拴紧并保持在循环中的人，另一类是指挥代理大军的人。我属于前者，也许是因为我老了。尽管我尽了最大努力，除了研究任务外，我从未能让代理大军为我工作。让它们编写大量代码对我来说是一场灾难。也许秘诀就是不看它们生成的代码，并祈祷一切顺利。我不觉得我在循环中会比那些拥有代理大军的人效率低。我还想指出，并没有很多“代理大军”流派的人真正发布了他们的作品，而我则尽量开源我所有的东西并记录下来。所以也许这里面有什么教训。

在我深入介绍几个精选的“AI”项目之前，我认为我对一年来使用编码代理的总结是：**还没有人知道如何正确地做到这一点**。我们都只是在把屎扔到墙上，宣布胜利，同时暗地里为我们要代理疯狂运行而在代码库中引入的所有技术债务哭泣。如果有人声称他们找到了解决方案，不要相信他们，他们没有。但这似乎有一些原则正在结晶，可以带来更可预测的结果，即使这些结果还远不可靠。

### Texty

**Texty** 建立于 2025 年 1 月，在当前的编码代理形式出现之前，它是我的第一个完全“凭感觉编码”（vibe coded）的项目。我有阅读障碍，所以我的语法和拼写非常糟糕。我想要一个简单的工具，可以在我用来写文本的应用程序中工作：浏览器和移动应用程序。

Texty 所做的只是在任何获得焦点的文本编辑字段旁边显示一个小按钮。点击它，它就会提取文本，通过 LLM 进行拼写和语法检查，然后将更正后的文本放回字段中。我将其构建为浏览器扩展和 Android 应用程序。Android 版本使用辅助功能服务（Accessibility Service），因此它可以观察所有正在运行的应用程序，检查其 UI 并与 UI 组件交互。

我用了几个月，直到我厌倦了这个工作流程。它太繁琐了，而且辅助功能服务让应用程序明显变慢。

对我来说，构建 Texty 的真正好处是它是我第一次涉足“代理工程”（即让 LLM 掌舵）。在 Texty 之前，我只使用 Cursor 进行标签补全，有时用于重写函数。有了 Texty，我想看看 LLM 在极少人工干预的情况下能构建一个项目走多远。虽然我不得不手动修复一堆东西，但扩展和应用程序的简单性使 LLM 实际上成功构建了我想要的东西。我没有看过极其复杂的辅助功能服务 API，我希望我永远也不用看。

### 黑掉 Claude Code (Hacking Claude Code)

大约在四月份，我开始使用 **Claude Code**。我对它的实际工作原理感到好奇。所以我做了任何理智的人都会做的事情：我修补了二进制文件以禁用其反调试检查，对 `fetch` 进行了猴子补丁（monkeypatch）以拦截所有 API 调用，并开始四处探查。

一些有趣的发现（针对当时的 Claude Code 版本）：

* Claude Code 使用 Haiku 模型来生成那些异想天开的“请稍候”消息。针对. 你. 输入的. 每一个. token。
* 它还使用 Haiku 来检查 bash 命令是否包含注入攻击。一个 LLM 判断另一个 LLM 的输出是否危险。这能出什么错？
* `/cost` 命令对 Max 计划用户禁用，因为显然我们不需要知道数字。数字对我们不好。我也修补了那个。
* 系统提示告诉 Claude 清理测试文件，这解释了为什么它总是删除它不应该删除的东西。他们后来移除了这个。
* 他们在不同的工具中三次添加了 emoji 限制，因为每个人都讨厌 LLM 到处注入 emoji。

我在两篇博客文章中写下了全部细节：[cchistory: Tracking Claude Code System Prompt and Tool Changes](https://mariozechner.at/posts/2025-04-24-hacking-claude-code/) 和 [Patching Claude Code for debugging and /cost support for Max users](https://mariozechner.at/posts/2025-05-01-patching-claude-code/)。我还建立了 `cchistory.mariozechner.at`，在那里你可以看到任何两个 Claude Code 版本的系统提示和工具定义之间的差异。

在 Claude Code 拥有钩子（hooks）和其他扩展功能之前，我构建了一堆工具来帮助我的工作流程：`claude-trace` 用于记录所有 API 请求并查看 Claude 隐藏的所有内容（系统提示、工具输出、原始 API 数据），`claude-bridge` 用于在 Claude Code 中使用其他 LLM 提供商，`cc-antidebug` 用于禁用反调试并为 Max 用户启用 `/cost`，`cccost` 用于精确的实时成本跟踪（反转：你不能仅根据会话 .jsonl 文件准确跟踪成本），以及 `claude-notify` 用于在 Claude 需要输入时获取系统通知。

此后我已经放弃了 Claude Code，并编写了自己的编码代理利用程序（harness），在过去的几个月里它一直为我服务。稍后会详细介绍。

### 探索并放弃 MCP (Exploring and Abandoning MCP)

我可能会失去我的黑客信誉，但我实际上喜欢用 VS Code 进行代码探索、查看差异和其他导航任务。在使用 Claude Code 时，我希望它能够在工作时为我在 VS Code 中打开、突出显示和比较文件。早在四月份，还没有 Claude Code VS Code 扩展。当时让 Claude Code 与 VS Code 交互的唯一方法是构建一个 MCP 服务器。所以我构建了 **vs-claude**，一个带有集成 MCP 服务器的 VS Code 扩展。这允许我机器上的任何 Claude 实例访问任何 VS Code 窗口，打开文件，突出显示行并显示差异。它工作得还行。

构建该扩展和相关的 MCP 服务器让我很好地掌握了什么是可能的。所以自然地，我开始构建更多的 MCP 服务器，比如用于 Gmail 集成的 `mailcp`，总的来说这简直太糟糕了。它基本上只是向代理的上下文中拉了一堆 JSON。虽然它能用，但这绝对远非理想。

这也是让我意识到 MCP 服务器输出只能由 LLM 在其内部上下文中进行转换、过滤或组合的项目。这太糟糕了，特别是与通过其 bash 工具链接多个命令可以实现的组合性和上下文空间保留相比。

意识到这一点后，我开始编写更多 CLI 工具，LLM 可以调用这些工具并通过 bash 轻松组合。最初的尝试是 `lsp-cli`，它使用语言服务器协议（LSP）从代码库中提取符号信息。我在一个棕地（brownfield）项目中将其作为基于代理的工作流程的一部分使用，效果相当不错。但这并不是 LLM 直接调用它。它是一个小 CLI，为 LLM 准备上下文以供其工作。如果你想了解更多，请查看 [Prompts are Code](https://mariozechner.at/posts/2025-06-18-prompts-are-code/) 博客文章。

总的来说，我并没有发现 LSP 服务器对 LLM 有多大帮助，无论是代码探索还是报错。我发现设置强制类型检查、linting 等作为提交钩子的一部分，对于让代理修复其错误要有效得多，而且不会用可能无关的 LSP 诊断填满你的上下文窗口。

在 2025 年夏天，AI 社交媒体上到处都是 MCP 的讨论，这与我自己使用 MCP 和 CLI 工具的经验不符。这促使我去看看在当前一代代理中到底什么表现更好。我试图通过一个非常不科学的评估来量化这一点。结果是：对于许多用例，无论是 MCP 还是 CLI 并没有太大关系，特别是那些组合性不起作用的用例。更重要的是，无论你使用什么工具，它都不会向你的上下文中拉一吨的 token。由于 CLI 工具更容易创建、维护和调试，我最终放弃了所有 MCP 服务器，并在 [What if you don't need MCP](https://mariozechner.at/posts/2025-07-16-what-if-you-dont-need-mcp/) 中分享了我的结论。

### Yakety

我也建立了一堆 Google CLI 工具，以便我的代理可以访问 Gmail (gmcli)、日历 (gccli) 等等。

我还构建了一个语音听写工具 **Yakety**。它会在按下快捷键时开始录音，松开时通过 API 发送录音，并将转录内容粘贴到当前获得焦点的任何应用程序中。

我最初计划将其作为一个商业产品，实行一次付费终身更新的政策，包括源代码。但随着我被其他事情分心，这个想法就逐渐消失了。此后我停止使用语音听写与我的编码代理一起工作。大多数时候，我需要指定文件路径和其他东西来构建上下文，而语音转录不太适合这种工作。而且，我打字非常快。但我至今仍在使用 Yakety，只是不用于编码。我主要用它来写作，就像写这篇博客文章一样。这是我写这篇博客前一个半小时的 [YouTube 视频](https://www.youtube.com/)。

它像其他的那么花哨吗？不。但我确切地知道它在做什么，而且从今天开始它是开源的，可能充满了代码腐烂。请享用！

### VibeTunnel

六月的某个时候，Peter 邀请 Armin 和我去他位于维也纳的漂亮公寓，我们打算在 24 小时内建立一个东西。起初我们没有任何想法，所以我们只是对着白板开始抛出想法。唯一坚持下来的想法是：我们都希望能够在旅途中通过手机控制 Claude Code（那时我们都使用的编码代理）。市场上真的没有类似的东西，无论是作为产品还是开源项目。所以我们想，如果我们三个人坐在一起 24 小时，让无数个 Clawds 为我们工作，这能有多难。于是 **VibeTunnel** 诞生了。

整件事是一次即兴“凭感觉编码”（vibe coding）的实验。我们像臭鼬工厂团队一样工作，每个人分配项目的不同部分。细节很模糊，但我记得 Armin 负责后端，Peter 负责专用的 macOS 应用程序，我负责前端，这是我最不擅长的东西。当 Peter 和 Armin 开发各自的部分时，我拼凑了一个后端，以便我的前端有一些东西可以工作，特别是终端流。Armin 自然是用 Rust 编写后端，结果发现 Node.js 版本实际上工作得更好。我们最终得到了一个能工作但基本上无法维护的弗兰肯斯坦怪物。

尽管如此，这是一个非常有趣的实验。我们三人在现实生活中度过了一段美好的时光，因为通常我们只是在网上闲聊。在我们解散后，Peter 试图保持 VibeTunnel 运行，将其变成一个开源项目，但那时该项目可能太复杂了，没人能上手。Peter 此后创建了 [Clawdis](https://clawdis.com/)，这是 VibeTunnel 的一个更好的版本，可以直接与 WhatsApp 和其他聊天应用程序集成。它也是一艘宇宙飞船，我还没敢看它的代码库，但似乎比 VibeTunnel 工作得好得多，至少对他来说是这样。

### Sitegeist 和 Cellgeist

十月的某个时候，我开始开发一个名为 **Sitegeist** 的浏览器扩展。它的目的有双重。一方面，它是一个深度研究代理，允许我通过直接使用 Google 和与网站交互来研究公司和人员或查找特定主题的信息。

它也是一次性抓取的绝佳伴侣，我不必为此自己编写程序。如果你喜欢，你也可以用它来购物，这是 Google 和所有其他公司总是演示的用例。我认为那是对技术的完全浪费。浏览器代理有更好的用途。

与 Chrome DevTools Protocol 或 Playwright 之类的东西相比，好处是我获得了研究过程的完整跟踪，因此我可以更容易地验证研究。Sitegeist 具有完整的会话管理功能，如果你愿意，可以返回并继续研究会话。它还支持多提供商、多模型，因此你可以使用任何类型的模型。根据经验，Claude 模型目前的表现最好。

Sitegeist 还带有 artifacts（工件），你可能从 Anthropic 的 Web UI 或 OpenAI 的 Canvas 功能中了解这一点。Sitegeist 可以创建内存中的 HTML、Markdown、Excel、PowerPoint、Word 文档等，我也可以下载这些文档。这让 Sitegeist 可以创建包含研究结果的交互式仪表板，我可以对其进行迭代。

为了更好地自动化你经常访问的网站，Sitegeist 配备了一个技能系统。

我还做了一个有趣的小实验，构建了 **Cellgeist**，它完全是一回事，只是生活在你的 Excel 中。这也是我要开源的，但这实际上只是一个概念验证，而 Sitegeist 实际上已经相当强大和有用了。

### pi 编码代理 (pi coding agent)

Anthropic 不断向 Claude Code 添加功能，并经常搞乱工具集和系统提示。这搞砸了我的工作流程，几个月来我变得越来越恼火。所以我最终着手编写我自己的小编码代理，完全按照我想要的方式工作。这意味着剥离所有我不需要且我认为是反模式的东西。没有 MCP 支持，没有内置子代理，没有计划模式，没有待办事项工具。我发现如果只给模型一套最少的工具：读取、写入、编辑、bash，它们实际上非常有效。除此之外的任何东西都应该是这个最小核心的扩展，而不是内置的东西。正如 Armin 所说，**pi** 是“黑客的开源选择”。它可以无限扩展而无需触及核心本身，允许你创建完全不同的代理，比如 Peter 的 Clawdis。

虽然核心是极简的，但 pi 配备了你习惯于从 Claude Code、Codex、Amp、opencode 等获得的所有舒适功能。只是你可以轻松地在其之上编写自己的功能，甚至自定义压缩。这也包括钩子和自定义工具能够为 pi 提供自定义 UI，这是我在任何其他编码代理框架中都没见过的功能。

pi 也是多提供商和多模型的，所以你可以使用你的 GitHub Copilot 订阅、Claude Max 计划、Cerebras、OpenRouter、本地模型等等。你也可以在同一会话中切换模型，这可能是个好主意，也可能不是。它还包括成本跟踪，这是其他编码代理框架做得很差的地方。

我已经在 [What I learned building a minimal LLM coding agent](https://mariozechner.at/posts/2025-10-01-building-pi/) 中详细写了我是如何以及为什么要构建 pi 的，所以我不会在这里重复。自从我写了那篇博文以来，pi 已经收到了很多好心人的第三方贡献。

我有点高兴我带着它回到了开源游戏。在每个人都睡着后的深夜坐下来，在 GitHub 上与人合作或调整我发现不足的小细节，这给我带来了巨大的快乐。在过去的两个月里，Pi 一直是我唯一的日常驱动工具。

### 教我的语言学家妻子使用 Claude Code

我可爱的另一半 Steffi 是一位语言学家，通常在 Excel 中手动进行数据分析。虽然她有一点技术背景（一点 HTML、JavaScript 和一些 Python），但她绝对不是程序员。当她做上一篇论文时，我从她肩膀上看她在做什么，并意识到 Claude Code 可以帮助她完成其中一些工作。所以我们用了两个晚上，分两次简短的会议教她使用 Claude Code。

她的研究项目涉及分析 32 位演讲者的奥地利方言动词前缀。她有装满 Excel 文件的文件夹，包含 18,000 多行语言注释。手动工作流程将是：打开每个文件，将包含由分号分隔的四个值的一列拆分为四个单独的列，过滤掉标有星号的某些形式，然后按演讲者和设置汇总结果。

取而代之的是，Claude Code 为每一步编写了 Python 脚本。一个脚本将所有 Excel 文件转换为具有正确分隔符的 CSV。另一个将注释列拆分为四列。第三个过滤掉不相关的形式并将它们写入单独的文件。然后是分析脚本，统计每位演讲者和设置的三种可能的语言现象值，随后是比较设置和分析因素的汇总统计数据，聚类分析，最后是一个生成图表的可视化脚本。

这个工作流程的美妙之处在于可重复性。当我们发现 18,000 个数据点中有 10 个在源数据库中有错误时，她在云数据库中修复了它们，重新导出，然后重新运行这三个 Python 脚本。一切都会自动重新生成。她可以将原始数据和脚本作为补充材料附在论文中，任何人都可以重现她的结果。

我觉得有趣的是，仅仅通过告诉她 Claude 可以为她编写代码，她就很快建立了一种关于什么是可能的、什么是困难的直觉。通过给她一些关于如何构建工作的想法（构建由小脚本组成的管道，这些脚本将文件作为输入并生成作为下一个小程序输入的文件），她几乎立刻就能自己使用 Claude Code 了。她无法判断 Claude 为她生成的 Python 代码。但她可以判断每个管道阶段的输出并验证其正确性。

这是代理的一个重复主题：**它们只有在领域专家手中才真正有效**。另一个重复的主题是，领域专家还不了解这些工具如何真正帮助他们自动化可验证的任务，以便他们有更多时间专注于实际工作。虽然与 Steffi 的实验在提高她的生产力方面取得了巨大成功，但我仍然很难想出如何向非技术领域专家传授这种思维方式。

## 调查野生垃圾内容 (Investigating Slop in the Wild)

这就是这种“工作”的有趣之处。以下是我过去一年做的一些有趣的小探索。

### 奥地利媒体 LLM 使用情况

Commander Raika 在 Bluesky 上发布了一张截图，显示 OE24 的一篇文章包含未经编辑的 ChatGPT 伪影。这就是我在行业里所说的“被极客狙击了”（nerd sniping）。

奥地利的报纸得到了大量补贴，既有政府广告也有直接资助。其中一个资助项目叫做“Fonds Digitale Transformation”（数字转型基金），报纸可以提交粗制滥造的项目描述，列出数字化工作，然后获得数千到数十万欧元。这些项目大多只是胡扯。所以当我看到 Commander Raika 的截图时，我想这是一个调查这笔钱是如何花的好时机。

那么，我如何通过数据调查 OE24 是否使用 LLM 撰写文章呢？我们已经有了直接证据，因为他们在在线文章中留下了 ChatGPT 输出的痕迹。但这可能只是报社的一个人搞砸了。我需要更多数据。

基本思路是这样的：LLM 喜欢到处插入破折号（em-dashes）和连接号（en-dashes）。假设是在引入 ChatGPT 之后（加上技术实际到达新闻编辑室的一段时间），我们会看到破折号或连接号的使用随时间增加。这种增加显然也可能有其他原因，例如风格指南的变更、新作者加入团队或 CMS 的变更。但如果有足够的数据，我们或许能够区分这些原因。

所以我抓取了 OE24 的网站地图，下载了 2021 年到 2025 年每年 10 月的数千篇文章，并统计了按文章长度归一化的破折号。结果：2021-2024 年破折号使用稳定，然后在 2025 年出现大规模激增。像购物、动物保护和生活方式内容等类别从几乎没有破折号变成了到处都是。

由于激增并不是在所有类别中均匀分布的，我认为这并非风格指南或 CMS 变更导致的。同样不太可能的是，他们将这些类别的整个团队换成了喜欢用更多破折号的作者。所以我认为这是直接证据，表明至少对于这些类别，OE24 现在让 LLM 撰写文章。这很有道理，因为这是 LLM 擅长的内容，不一定需要人在环。我制作了一个 YouTube 视频并写了一个 Bluesky 帖子解释了整件事。

在第二次调查中，我的语言学家朋友 Jenia 加入了我。我们扩展到了 Exxpress、Heute、Krone 和 Falter。Falter 和 Krone 也显示出破折号使用率上升。

虽然我们的偏见会让我们相信这对 Krone 来说是意料之中的，但对于通常被认为是高质量周刊的 Falter 来说，我们没想到。他们也开始使用 LLM 了吗？

不一定。许多 CMS 系统会自动将 `<空格><连字符><空格>` 转换为 `<空格><连接号><空格>`。所以 CMS 更新可以解释这种模式。解决方案：差异分析。如果破折号上升而连字符成比例下降，那就是 CMS 对现有内容进行了查找和替换。如果破折号爆炸式增长而连字符保持平稳，那就是添加了新的大量破折号的内容，这意味着 LLM 生成。Falter 和 Krone 显示了 CMS 模式。

Exxpress、OE24 和 Heute 显示了 LLM 模式。

在最后的调查中，我们添加了 Kurier 和 Kleine Zeitung。两者都显示在特定垂直领域（如生活方式和娱乐）选择性地采用 LLM，而硬新闻则保持人工撰写。

这与 RTR（奥地利监管机构）的公共资金非常吻合。OE24 的母公司在 2024 年因“AI 图像识别/数据库/校正/创作”获得了 287,000 欧元。Heute 因“AI 支持的新闻业”获得了 140,876 欧元。奥地利纳税人资助了这些 AI 部署，但这几家出版物都没有向读者披露 LLM 的使用情况。

作为副产品，我现在拥有一个针对所有主要奥地利新闻网站的爬虫。我正致力于将其转变为媒体监控工具和涵盖过去 30 天发布的所有新闻文章的公共搜索引擎。

为了好玩，Jenia 和我让 Claude 写了一篇完整的方法论论文，包含了所有图表和细节，如果你想深入挖掘的话。

### fobizz 学校 AI 工具

LLM 进入学校和大学现在已成为现实。这究竟是好是坏还有待观察。政治决策者最喜欢做的莫过于将 AI 工具塞进学校，假装它们提高了一些编造的指标。我真的很讨厌那个。我相信如果使用得当，LLM 实际上可以帮助学习。但在野外看到的东西只是[简单的封装，指向]免费的 GPT API 端点。

### 招聘中的面部识别论文

有人向我发送了一篇声称可以使用 AI 根据面部特征预测个性的论文，并询问这是否正在生产中使用。我读了那篇论文，那是**方法论上的垃圾**。

该论文试图通过引用遗传学研究来建立面部特征与个性特征之间的联系。但引用的论文只说个性特征部分是遗传的。它们**没有**说你可以从面部特征推断个性特征。作者随后建立在之前一篇已经在《Cell》杂志上被斥为“机器学习伪科学的复活”的论文的方法之上。

数据来自 Revelio，一家使用 AI 生成大部分数据的公司。薪资数据是模型生成的。“种族”分类基于姓名和地点，加上图像识别。他们使用了一个在自我报告的个性特征上训练的模型，任何心理学家都会告诉你这是不可靠的。

该论文甚至承认其预测能力很弱（低 R²），但声称该方法是“新的”。稳健性检查很可笑：他们在 28 名女性和 37 名男性身上测试了“冷静”与“快乐”的面部表情，所有人的光线和发型都完全相同。在任何现实世界条件下都没有变化。

该论文问“筛选掉那些面孔预测不太理想个性的人是否合乎道德”。这不是一个具有挑战性的问题，因为这种方法不起作用。具有挑战性的问题是，为什么像这样的伪科学会在《经济学人》等媒体上获得报道。这是披着 AI 新外衣的颅相学。

### 其他调查

* **政府聊天机器人**：我使用 Sitegeist 在奥地利政府网站上查找并测试聊天机器人。Kremsi (Kremsmünster) 是由 schorn.ai 构建的，很容易越狱。其隐私政策声称仅在欧盟处理数据，但实际上托管在美国的 Vercel 上。它赢得了行政奖。Feldi (Feldkirchen) 由 AI Concierge 构建，该公司与 AMS Berufsinfomaten 是同一家公司。KLARA (Klagenfurt) 使用同一平台。Telfer Copilot (Telfs) 托管在爱尔兰的 Azure 上，实际上可以阅读市议会协议，这确实很有用。我做了一个 YouTube 直播展示了整个过程。
* **Clinara 医疗转录**：有人发布了关于 Clinara 的帖子，这是一家向医生销售 AI 转录的奥地利初创公司。我通过阅读一篇标准德语文章并分析单词错误率来测试它。基于 Whisper 的转录大约有 5-10% 的单词错误率，这意味着每 100 个单词中大约有 5-10 个是错误的。对于医疗文档来说，这令人担忧。奖励：他们的网络应用程序从客户端发送系统提示，所以我可以使用他们的服务器作为免费的 GPT API 端点。
* **StoryOne 书籍生成器**：FAZ 发布了一篇关于 StoryOne 的软文，这是一家奥地利初创公司，声称用 AI 彻底改变非虚构书籍写作。我做了一个自测：给它一篇我的博文，让它生成一本“基于事实”的书。结果是 75 页（其中 18 页是空白的“为了布局”），尽管他们声称是“神经符号 AI”，但实际上没有事实充实，只是幻觉般的 LinkedIn 垃圾内容。所有相关方都应该感到羞耻，包括 FAZ。

## 工作坊

2025 年也是我做更多工作坊的一年。在 2024 年，我创建了一个关于如何将 LLM 集成到你的程序中的工作坊。今年，我为普通人创建了一个生成式 AI 工作坊，解释 LLM 的来龙去脉、聊天界面等等。目标始终是灌输对黑盒内部实际发生情况的理解。这意味着该工作坊实际上是对 LLM 及其利用框架机制的深入研究。参与者将学习 LLM 实际上如何处理他们上传的 PDF，视觉如何工作以及有哪些限制等等。这基本上是一个去神秘化和现实检查，应该有助于参与者在使用这些工具和理解它们的故障模式时更有效。

Profil（奥地利周刊）的记者 Max Müller 问我是否想在他们的神圣大厅里举办这个工作坊。所以我去了。与调查记者一起研究这些材料并深入了解他们的日常工作是非常有益的。

12 月，我还创建了一个代理编码工作坊，部分基于更通用的 GenAI 工作坊，配有许多小演示和注意事项，而不是规定性的。这基本上是我过去一年经验的 3-4 小时浓缩版。

## 结论

像往常一样，这篇年度回顾有点长。相信我，还有更多东西可以填进去。

我今年最大的收获是，我真的很享受在现实生活中见到我的同行。对于 2026 年，我希望能参加更多的聚会，结识更多有趣的人。

在学习方面，我对我的电子学知识水平感到非常高兴。我已经危险到可以制造能杀死我的东西了，这太棒了。至于软件工程，我相信我很好地跟上了代理编码领域，尽管我尽量不喝太多的迷魂汤（Kool-Aid）并保持脚踏实地。

我的生产力通过 LLM 提高了吗？我真的不知道。与“两年回顾”相比，你会发现并没有多出那么多项目。区别可能在于我今年从事的项目具有更多的技术深度。如果没有编码代理，我可能因为没有时间手动完成所有工作而不会开始这类项目。

但这又只是感觉。我认为这就是今年的总体主题。**一切都只是感觉 (It's all just vibes)。**
